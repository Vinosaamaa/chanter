# Issue #192 Change Log — SEC-12: Harden local infra binds and credentials

## Problem

Docker Compose published all infra ports on `0.0.0.0` (all interfaces), making Postgres, Redis, Redpanda, MinIO, and LiveKit reachable from any network interface on the developer machine. Additionally:

- Redis had no password (`requirepass` not set).
- MinIO shipped with fixed weak credentials (`chanter` / `chantersecret`) hard-coded in `.env.example`.
- LiveKit ran with `--dev` flag and `devkey:secret` as public known defaults.
- `LIVEKIT_API_KEY` / `LIVEKIT_API_SECRET` defaulted to `devkey` / `secret` in `lib.sh` when not set.

## Changes

### `infra/docker-compose.yml`

- All `ports:` entries now bind to `127.0.0.1:<host_port>:<container_port>` — covers Postgres, Redis, Redpanda (Kafka + Pandaproxy), MinIO (API + console), realtime-service, and LiveKit (TCP + UDP).
- Redis: added `--requirepass ${REDIS_PASSWORD}` and updated healthcheck to pass `-a ${REDIS_PASSWORD}`.
- MinIO: removed `:-chanter` / `:-chantersecret` fallback defaults so empty `.env` values fail fast rather than silently using known credentials.
- LiveKit: removed `--dev` flag; `LIVEKIT_KEYS` env var format preserved (`"${LIVEKIT_API_KEY}: ${LIVEKIT_API_SECRET}"`).
- realtime-service container now receives `REDIS_PASSWORD: ${REDIS_PASSWORD}` env var.

### `.env.example`

- `REDIS_PASSWORD` added as empty placeholder with comment (SEC-12).
- `MINIO_ROOT_USER` / `MINIO_ROOT_PASSWORD` changed to empty placeholders (removed hard-coded `chanter` / `chantersecret`).
- `LIVEKIT_API_KEY` / `LIVEKIT_API_SECRET` changed to empty placeholders (removed hard-coded `devkey` / `secret`).

### `scripts/product/init-env.sh`

- Added generation of `REDIS_PASSWORD` (32-char random alphanumeric) when empty.
- Added generation of `MINIO_ROOT_USER` (`chanter-local`) and `MINIO_ROOT_PASSWORD` (32-char random) when empty.
- Added generation of `LIVEKIT_API_KEY` (`chanterlocal`) and `LIVEKIT_API_SECRET` (48-char random) when empty.
- Refactored `needs_replace` to separate empty-value check from the secret-length/forbidden check.

### `scripts/product/lib.sh`

- `product_validate_runtime_secrets`: added checks that `REDIS_PASSWORD`, `MINIO_ROOT_USER`, `MINIO_ROOT_PASSWORD`, `LIVEKIT_API_KEY`, and `LIVEKIT_API_SECRET` are non-empty. Any missing value prints a clear error and aborts `make product-up` / `make product-supervise`.
- Removed hard-coded `LIVEKIT_API_KEY=devkey` / `LIVEKIT_API_SECRET=secret` fallback defaults from `product_load_env`.

### `Makefile`

- Added `require-infra-secrets` guard (mirrors existing `require-jwt-secret`) that checks `REDIS_PASSWORD`, `MINIO_ROOT_USER`, `MINIO_ROOT_PASSWORD`, `LIVEKIT_API_KEY`, `LIVEKIT_API_SECRET` before `infra-up`.

### `backend/realtime-service/src/main/resources/application.yml`

- Added `spring.data.redis.password: ${REDIS_PASSWORD:}` so the Java realtime-service authenticates to Redis with the password. Empty string is the Spring Boot sentinel for "no password" — if `REDIS_PASSWORD` is unset it falls back gracefully, but `product_validate_runtime_secrets` prevents startup with an empty value.

### `docs/operations/getting-started.md`

- Step 2 ("Create your local config") now lists all secrets generated by `make product-env`, including the new SEC-12 secrets.
- Added a **Shared / staging note** that warns against copying local `.env` to shared hosts and documents the 127.0.0.1 bind policy.

|---|---|---|

## Breaking local notes

- **Existing `.env` files** that still have `LIVEKIT_API_KEY=devkey` / `LIVEKIT_API_SECRET=secret` will now fail validation (those values are empty in `.env.example` but not forbidden by name — the check is just non-empty). If you have an existing `.env` with `devkey`, it will still pass validation since it is non-empty. Run `make product-env` to regenerate if preferred.
- **Existing `.env` files** with `MINIO_ROOT_USER=chanter` / `MINIO_ROOT_PASSWORD=chantersecret` will still work (they are non-empty). Run `make product-env` only if you want to rotate them.
- **Existing MinIO volume** (`minio-data`) stores data with the old credentials. If you change `MINIO_ROOT_USER` or `MINIO_ROOT_PASSWORD` while the volume exists, MinIO will reject the new credentials until the volume is cleared: `docker compose -f infra/docker-compose.yml down -v` (destroys all local data). For local dev this is acceptable.
- **Redis data** is unauthenticated in old containers. After adding `requirepass`, restart with `docker compose -f infra/docker-compose.yml down && make product-up`.

## Shared / staging guidance

Do **not** use any values from `.env.example` on a shared or staging host. Generate unique secrets per environment. All infra ports are bound to `127.0.0.1` — expose only via a reverse proxy (nginx/caddy) with TLS and auth.
